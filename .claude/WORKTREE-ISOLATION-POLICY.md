# Worktree Isolation Policy — BANXE AI BANK
# Rule ID: ISO-FIX-01 | Created: 2026-07-17
# Status: proposed (this file lands via PR from agent/factory/isofix01/worktree-isolation)

## Why this exists

A live audit on 2026-07-17 found 48 git worktrees registered against
`banxe-emi-stack`, ~35 of them dirty. Twelve of them were created **inside**
the main repo tree, at `/home/mmber/banxe-emi-stack/.claude/worktrees/agent-*`.
Nested worktrees defeat the purpose of a worktree (an isolated checkout for
one task): they live under the same directory `find`/`grep`/status tooling
walks for the main checkout, so their dirty state and their file contents
bleed into any tool that scans the main tree.

Separately, the factory itself was writing runtime state
(`.claude/memory/commit-log.jsonl`, `.claude/memory/services-map.md`, and
ad-hoc snapshot files) into tracked, non-ignored paths inside the repo. That
guaranteed the main worktree was *always* dirty, which trains both humans and
automation to ignore "uncommitted changes" warnings — the exact opposite of
what a clean-tree invariant is for.

This policy is the fix for both. See also `.claude/hooks/worktree-guard.sh`
for an advisory (not yet wired) script that can enforce rule 1 below.

## Rules

1. **Every task gets its own fresh worktree, created OUTSIDE the repo tree.**
   Use `/home/mmber/wt/<id>` or `/home/mmber/banxe-<slug>` (sibling to the
   repo, never a subdirectory of it). Branch from `origin/main`. Name the
   branch per ADR-060: `agent/<role>/<id>/<slug>`, where `<role>` is one of
   `central|right|factory|specproj`, `<id>` is alnum with no hyphens, and
   `<slug>` is lowercase kebab-case.

   ```bash
   git worktree add /home/mmber/wt/<id> -b agent/factory/<id>/<slug> origin/main
   ```

2. **Never create a worktree inside `.claude/worktrees/` or anywhere else
   under the repo root.** A worktree nested inside the repo it was branched
   from pollutes `git status`, `grep -r`, `find`, and any editor/tool that
   walks the main tree from its root. If you find yourself about to run
   `git worktree add .claude/worktrees/...` or similar — stop; that path is
   inside the repo tree and violates this policy.

3. **The working tree must be clean before a task starts.** Run
   `git status --porcelain` in the target worktree before doing anything;
   it must be empty. If it isn't, resolve that first (commit, stash, or
   escalate) rather than layering new changes on old residue.

4. **The task must leave no residue when it finishes.** Before ending a
   task, either commit the change (on that task's own branch) or explicitly
   stash it with a labeled `git stash push -m "<id>: <reason>"`. Never leave
   uncommitted, unstashed changes behind for the next task to trip over.

5. **Factory runtime memory lives in gitignored paths only.** State that is
   generated by the factory as a side effect of running (session logs,
   commit-log, generated service maps, local docker-compose overrides) must
   never be a tracked file. See the `.claude/memory/` and
   `docker/docker-compose.*.override.yml` entries in the repo's
   `.gitignore` (added alongside this policy). If a new category of
   factory-runtime file appears, gitignore it before it is ever committed —
   don't commit-then-regret.

## What this policy does NOT do

- It does not retroactively clean up the 12 existing nested worktrees or the
  ~35 dirty worktrees found in the 2026-07-17 audit. Those require a
  deliberate, per-worktree migration decision (real uncommitted work must be
  rescued before removal) — see the ISO-FIX-01 inventory report for the
  per-worktree recommendation.
- It does not auto-enforce anything. `.claude/hooks/worktree-guard.sh` is an
  advisory script an operator can wire into `settings.json` after review; it
  is intentionally not self-installing.
